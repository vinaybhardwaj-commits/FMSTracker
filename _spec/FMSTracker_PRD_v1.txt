1. Vision & Scope

1.1 What FMSTracker is
FMSTracker is a mobile-first PWA used by EHRC’s facilities maintenance crew (led by
Charan Kumar, Facilities Manager) to inspect, log, and prove the upkeep of every critical
hospital facility and environmental system. It encodes the EHRC FMS Library — every
preventive-maintenance task across fire, MGPS, electrical, water, HVAC, lifts, plumbing,
building, signage, security, hazmat/BMW, and statutory compliance — into a
daily/weekly/monthly/quarterly/annual cadence the app surfaces, claims, completes, and
proves with photos and selfies.

The unit of value is continuous evidence of facilities operation — not a one-shot audit.
Every task completed in FMSTracker generates a timestamped, photo-backed record that
doubles as NABH FMS evidence. Failures are caught before they become breakdowns;
statutory renewals are caught before they become violations.


1.2 What FMSTracker is not (v1)
   Not a biomedical equipment tracker (Kiran’s domain — separate workstream).

   Not a housekeeping/sanitation tracker (covered by SanitizeTrack module in EHRC-Daily-
   Dash).

   Not an authenticated app (no logins; identity proven via selfies + photos).

   Not an AMC vendor portal (vendors continue to sign paper service reports; helper
   photographs them).
   Not a multi-site app in v1 (EHRC Race Course Road only).

   Not offline-capable in v1 (hospital has high-speed WiFi everywhere; pure online
   operation).


1.3 Spin-off rationale (vs. extending SanitizeTrack)
   Different actor: Facilities crew, not housekeeping staff.

   Different cadence model: Daily/weekly/monthly/quarterly/annual + statutory renewal —
   not AM/PM/NIGHT shifts.

   Different failure model: A missed extinguisher refill or DG load test is patient-harm +
   regulatory exposure; a missed HK task is recoverable.

   Statutory compliance as first-class object: Licences with renewal dates and lead-
   time tiers don’t fit the recurring-task pattern.

   Mutable register: SanitizeTrack hard-codes its seed; FMSTracker requires live admin-
   editable task templates because facilities reality changes (new equipment, vendor
   changes, NABH revisions).

2.1 First-launch flow (every device)
1. App opens → “Welcome to FMSTracker.”

2. “What’s your name?” → text input.
3. “Take a clear selfie. This is how others will know it’s you.” → camera capture.

4. Device gets a UUID (stored in localStorage ); name + selfie stored against it.

5. On task completion later, selfie is captured fresh (proof of presence) and matched
   visually against baseline (no biometric matching in v1 — humans can compare).


2.2 No-auth security model
   Identity = device UUID + baseline selfie + name.

   Every action is timestamped and tied to device UUID.

   Admin pages (CRUD on register, vendor edits, statutory edits) gated behind a 4-digit
   PIN that Charan + V know. PIN stored as bcrypt hash in env var.

   No PHI is stored; no patient data leaves the app. Photos are of equipment, vendors, and
   helpers only.




3. Core Concepts

3.1 Task Template vs. Task Instance
Task Template — the row in the register. Lives in task_templates table. Editable by admin.
Defines:

   What to do (name, acceptance criteria)

   Cadence (daily / weekly / monthly / quarterly / semi-annual / annual / statutory_renewal)

   frequency_in_days (1, 7, 30, 90, 180, 365)

   cadence_anchor (e.g., “every Monday”, “1st of month”, or fixed date for statutory)

   Actor type (in_house / amc_supervised / statutory)

   Evidence required (selfie+photo / selfie+photo+reading /
   selfie+vendor_report+next_due_date)

   Priority weight (1–100)

   NABH standard reference

Task Instance — a specific occurrence of a template, generated by the engine. Lives in
task_instances table. Has:

   Reference to template_id (snapshot semantics — see §3.4)
   due_date

   status: pending | claimed | done | skipped | overdue

   claimed_by_device, claimed_at, claim_expires_at

   completed_by_device, completed_by_name, completed_at

   selfie_url, photo_urls (array), reading_value, vendor_report_url, vendor_next_due_date

   skip_reason (constrained list)

   carryover_from_id (if rolled over from prior period)


3.2 Soft-claim mechanics
When two helpers see the same task on the list:

   Helper A taps “I’ll do this” → instance.claimed_by = device_A, claim_expires_at = now
   + 30 min.

   Helper B sees the task with grey-out: “Ravi is doing this — 24 min left” + Ravi’s avatar.

   If Helper A completes within 30 min → done.

   If Helper A doesn’t complete within 30 min → claim expires; task returns to open pool;
   anyone can pick it up.

   Helper B can manually release Helper A’s claim with a one-tap “Release” button +
   reason (“can’t reach Ravi”, “Ravi went home”).

3.4 Snapshot semantics (mutability handling)
When a task template is edited by admin:

    In-flight task instances (status = pending or claimed) keep their original snapshot of
    name, acceptance criteria, evidence required. They are NOT retroactively rewritten.

    Future task instances generated after the edit reflect the new template.

    An audit trail ( template_changes table) logs every edit: who (admin device + name),
    when, before/after diff.

This protects evidence integrity. If Charan edits “Inspect fire panel” to “Inspect fire panel
and verify battery voltage” mid-day, helpers who already see the simpler version on their
phone can complete it as-is. Tomorrow’s instance will reflect the new criteria.

Exception — emergency overrides: Admin can hit “Apply to all in-flight” with PIN re-entry,
for cases where a critical safety detail must propagate immediately (e.g., wrong disinfectant
listed). Logged as force_propagate event.

4. Functional Requirements
4.1 Worker (no-auth) flows
Today’s tasks page ( / ):

   Pinned banner: any RED or CRITICAL statutory items.

   Section: Overdue (carried over from yesterday — visually orange).

   Section: Today’s tasks (sorted by priority_weight desc, then by claim status).

   Each task shows: title, location, cadence badge, claim state (free / claimed-by-someone
   / mine).

   Tap task → task detail modal.

Task detail modal:

   Title, location, acceptance criteria (verbatim from template snapshot — “what done
   looks like”).

   Reference policy link.

   Big primary button: “I’ll do this” (if free) or “Complete this” (if claimed by me) or
   “Release” (if claimed by someone else).

   Skip button → reason picker → confirm.

Completion flow:

1. Tap “Complete this” → camera opens for selfie.

2. Selfie captured → next: “Take photo(s) of the work.”

3. If reading required → numeric input (with units hint from acceptance criteria).

4. If vendor-supervised → additional camera prompts for vendor presence + service report;
   expiry date input.

5. Submit → optimistic UI (“Saved ✓”) + background upload.

6. Task moves to “Done today” section at bottom.

Yesterday recap (always available):

   Show what got done yesterday vs. what was missed.

   “Who did what” with thumbnails — visible to whole crew, builds accountability.

6. Engine: Task Instance Generation

6.1 Generation cron
   Runs daily at 04:00 IST (Vercel Cron).

   Generates instances for today + next 7 days (rolling window — gives admin visibility
   into upcoming workload).
6.2 Generation rules

 For each active template in task_templates:
   if cadence == 'daily':
     generate one instance for today (if not exists)
   elif cadence == 'weekly':
     if today.day_of_week matches cadence_anchor (e.g., "every Monday"):
       generate one instance for today
     elif cadence_anchor == 'rotating floors' or 'rotating zones':
       cycle through floors/zones — generate one per week, advancing
   elif cadence == 'monthly':
     if today.day_of_month matches cadence_anchor:
       generate one instance for today
   elif cadence == 'quarterly':
     if today is in first month of quarter and matches anchor:
       generate one instance
   elif cadence == 'semi_annual' or 'annual':
     if today matches the anchor pattern:
       generate one instance
   elif cadence == 'statutory_renewal':
     NOT generated by cron.
     Instead: tier evaluated based on (current_expiry - today).
     Surfaced into today's list when entering 180/90/30 day window.



6.3 Carryover logic
   At 04:00 IST cron run, before generating new tasks:

      Find all pending or claimed instances with due_date < today .

      Mark them as overdue .

      These continue to surface at the top of today’s list (visually orange).

      They are NOT re-created as new instances — same row, just escalated state.

   If overdue >7 days, admin gets a flag in the audit log + dashboard counter.


6.4 Soft-claim cleanup
   Separate cron at every 15 minutes:

      For instances where status='claimed' AND claim_expires_at < NOW() :

          Set status='pending' , clear claimed_by_device , claimed_at ,
           claim_expires_at .
           Log to audit: claim_expired .

    Match EHRC brand: Arial; primary blue #0055FF ; dark navy #002054 ; white-dominant.

10. Storage & Performance (low-end phones)

10.1 Image compression
   Client-side: canvas-based resize to max 1024×1024, JPEG quality 0.7.

   Target: ≤150KB per photo.

   Selfies: max 512×512, quality 0.6, target ≤80KB.

   All compression happens before network upload.


10.2 Bundle size discipline
   No heavy date library (use native Intl.DateTimeFormat + small helpers).

   No charting library on worker pages (only on dashboard).

   Code-split admin routes — workers never load admin JS.

   Target: worker page initial load <200KB gzipped JS.


10.3 Network
   WiFi-only operation per spec; no offline mode in v1.

   Optimistic UI on completion: show “Saved ✓” immediately, queue upload in background.

   If upload fails: red toast + “Retry” button; instance reverts to claimed state.


10.4 Tested baseline
   Android 9+, 2 GB RAM, Chrome 100+.

   iPhone 7 / iOS 14+.

14. Open Questions (need V’s call before / during build)
1. Database choice — Vercel Postgres or Supabase? Daily Dash uses Vercel Postgres
     ( @vercel/postgres ); Rounds uses Supabase. FMSTracker can match either. Vercel
     Postgres is simpler for this scope but has tighter limits on connections. Recommend
     Vercel Postgres for v1 unless multi-site triggers earlier.

2. Admin PIN strategy — single shared PIN or per-admin PIN? A single shared 4-digit
     PIN is simplest but less auditable. Per-admin PIN means a small admin_pins table with
     named admins. Recommend single shared PIN for v1, switch to per-admin in v1.1 if
     Charan + V want individual accountability on register edits.

3. Statutory renewal lead-time triggers — should they generate distinct task
     instances? Current spec: tiers are visual states on the statutory item itself. Alternative:
     at 180d/90d/30d, generate three distinct task instances (e.g., “Fire NOC: start vendor
     coordination” at 180d, “Fire NOC: vendor visit booked?” at 90d, “Fire NOC: submission
     ready?” at 30d). Latter gives Charan explicit checklist actions; former is lighter.
     Recommend latter for high-stakes licences (Fire, KSPCB, AERB), former for lower-risk.
4. Naming — “Instance” vs “Round” vs “Card”? Internal data model says
